## 00:07 — 모든 프레임을 완벽하게
**source**: geeknews · **tags**: #UI #상태 #애니메이션 #전환 #사례
**url**: https://news.hada.io/topic?id=30467

- UI는 사용자가 앱 품질을 판단하는 거의 유일한 표면이며, 어느 순간 스크린샷을 찍어도 **화면 상태**가 말이 되어야 신뢰가 쌓임
- 완성도 높은 UI는 개발자가 다듬는 시간을 들였다는 신호가 되고, 코드 품질도 비슷하게 다듬었을 것이라는 **합리적 휴리스틱**이 됨
- 실제 기준은 화면 전환 사이 흰색 깜빡임, 부분 로딩, 로딩 중 재배치, 상태 문구 불일치, 부정확한 **애니메이션**을 없애는 것임
- 시작과 끝 상태가 좋아도 중간 프레임이 어긋나면 구성요소가 동기화되지 않은 느낌을 만들고, Photos 사례처럼 실제 변화가 없는 전환에서도 **거짓 감각**을 만들 수 있음
- 애니메이션은 전환 이해를 도와야 하며, 시작·끝뿐 아니라 그 사이 모든 프레임까지 관리해야 **정밀한 도구** 같은 UI가 됨

## 핵심 원칙

- Wayland의 명시 목표인 [“every frame is perfect”](https://wayland-book.com/protocol-design/design-patterns.html)는 현대 GPU 스택의 복잡성 속에서 제어권을 되찾으려는 기술적 목표임
- 같은 원칙은 UI에도 적용되며, 앱의 어느 순간을 스크린샷으로 찍어도 화면이 자연스럽고 일관되어야 함
- 사용자는 코드를 볼 수 없기 때문에 UI가 앱 품질을 판단하는 유일한 표면이 됨
- UI가 잘 다듬어져 있으면 개발자가 폴리싱 시간을 들였다는 신호가 되고, 코드도 비슷한 수준으로 다듬었을 것이라는 판단으로 이어짐

## 실제 기준과 사례

- 완벽한 프레임을 위해서는 화면 사이 **흰색 깜빡임**이 없어야 하고, 콘텐츠가 부분적으로 로딩된 상태나 로딩 중 레이아웃 재배치가 없어야 함
- UI 내부 상태는 일관되어야 하며, 한 부분이 “1 update available”이라고 말할 때 다른 부분이 “Checking for updates...”라고 말하면 안 됨
- 애니메이션은 자주 잊히며, 시작 상태와 끝 상태가 좋아도 그 사이가 어색하면 중간 스크린샷이 말이 안 되는 프레임이 됨
- Safari 사례에서는 플레이스홀더 텍스트가 가운데에서 움직이지만 커서는 왼쪽 위치에서 애니메이션되어 두 구성요소가 서로 동기화되지 않은 느낌을 만듦
- Photos 사례에서는 Crop과 Adjust 모드를 전환할 때 사진은 즉시 제자리로 이동하지만 크롭 테두리는 애니메이션되어, 모드 전환 중 무언가 미묘하게 바뀌는 **거짓 느낌**을 만듦
- 검색 UI 사례에서는 전환 이해를 도와야 할 애니메이션이 오히려 돋보기 움직임을 따라가기 어렵게 만듦
- Youtube 사례에서는 직사각형을 한 위치에서 다른 위치로 옮기는 단순한 작업에서도 이상한 움직임이 생기며, 이유와 관계없이 결과는 완벽하지 않은 프레임이 됨
- Preview 앱의 확대 애니메이션까지 포함해 핵심은 시작과 끝 상태뿐 아니라 그 사이 모든 순간에도 주의를 기울여야 한다는 점임

## 00:17 — 거절된 이모지 제안들
**source**: geeknews · **tags**: #제안 #문서 #항목 #범주 #결정
**url**: https://news.hada.io/topic?id=30468

- 공식 문자 제안서에 오른 뒤 새 문자 추가로 이어지지 않은 **이모지 후보**를 범주별 표로 모은 목록임
- 대상은 Unicode Consortium에 제출된 **공식 문자 제안서**의 이모지이며, 개요 문서의 논의나 비공식 요청은 집계하지 않음
- 공식 불승인 공지가 없어도 새 문자 추가로 이어지지 않은 제안은 거절로 간주되며, 목록은 **100% 완전하지 않을 가능성**이 높음
- 얼굴, 손동작, 음식, 동식물, 사람, 사물, 기호, 깃발, modifier 등 여러 범주에서 실제 제안 문서와 **decision** 항목이 연결됨
- 일부 후보는 승인 과정에서 다른 문자나 시퀀스로 대체됐고, 일부 문자는 Universal Character Set에 들어갔지만 **이모지 지위**를 받지 못함

---

## 목록의 범위와 판정 기준

- **공식 문자 제안서**에 포함된 이모지만 목록 대상으로 삼으며, 가능한 추가 항목을 논의한 개요 문서나 비공식 요청은 집계하지 않음
- 이모지로 쓰일 의도로 제안된 기호만 고려하지만, 일부 항목은 해당 지위가 불명확할 수 있음
- 한때 제안됐지만 다른 제안서나 약간 다른 형태로 최종 추가된 이모지는 목록에서 제외됨
- 공식 불승인 공지가 없더라도 새 문자 추가로 이어지지 않은 제안은 거절된 것으로 간주됨
- 목록은 완전하지 않을 가능성이 높으며, 실패한 이모지 제안의 더 완전한 열거는 Unicode Consortium의 Emoji Proposals Status 스프레드시트를 참조해야 함

## 표의 구성과 표시 방식

- 표는 **Sample, Name, Author, Date, Document, Decision** 열로 거절된 이모지 제안을 정리함
- 샘플 이미지는 대개 제안서에서 직접 가져오며, 항상 모범 사례를 정확히 반영하지는 않을 수 있음
- 제안 문서에 적절한 그림이 없으면 **sample** 열이 비어 있음
- **decision** 열은 해당 이모지의 거절과 관련된 문서와 action item을 나열함
- 거절 관련 논의는 매우 모호할 수 있으며, 일부 논의는 같은 제안서 안의 다른 문자만 다뤄 목록의 이모지에 대한 결정을 암시할 수 있음
- 제안의 존재만 인정하고 입장을 밝히지 않은 노트는 별표(\*)로 표시되며, 제안자에게 비공개로만 전달된 결정은 dagger(†)로 표시됨
- **withdrawn** 상태는 해당 이모지가 문서의 이전 반복본에만 있었고 이후 제안자들이 직접 대체 문서를 냈음을 뜻함

## 실제 영향

- ### 차등 개인정보보호의 위치

  - 차등 개인정보보호를 제거하면 비슷한 프라이버시 수준에서 유용성이 더 낮거나, 같은 유용성에서 프라이버시가 더 나쁜 기법만 남음
  - 경쟁 기법들도 노이즈 추가에 의존함
- ### 다른 기법들도 무작위성을 사용함

  - 다른 통계 기관에서 쓰는 [Cell Key method](https://sdctools.github.io/HandbookSDC/05-frequency-tables.html#sec-CKM_freq)는 통계에 노이즈를 더함
  - Census가 1990년부터 2010년까지 사용한 스와핑도 과정에 무작위성을 주입함
  - 표본추출은 통계 작업 전반에 널리 쓰임
  - [대체](https://en.wikipedia.org/wiki/Imputation_(statistics))도 기술적으로 데이터에 노이즈를 더함
- ### 일반화와 억제의 한계

  - 일반화와 억제는 매우 둔한 도구임
  - 일반화와 억제는 통계가 이미 매우 거칠고, 공개되는 통계 수가 많지 않은 상황에서만 작동함

  - 유용성 파괴는 특히 소수 집단에서 두드러짐
- ### 노이즈가 공격을 어렵게 만드는 이유

  - 통계 공개에 대한 프라이버시 공격은 [연립방정식을 푸는 문제](https://desfontain.es/blog/attacks-on-statistics.html)에 가까움
  - 모든 통계가 완벽히 정확하다고 확실히 알 때 이 작업은 훨씬 쉬워짐
  - 노이즈는 공격자가 확률을 계산하고, 불확실성을 정량화하고, 기준선을 신중하게 고려하도록 만듦
  - 무작위성은 공식 보장이 없어도 공개 회피에 유용하며, 공격을 훨씬 어렵게 만듦
  - 무작위성을 제거하면 공격은 사소해짐

## 왜 이런 일이 일어나는가

- 동기는 알 수 없음
- 목표가 향후 게리맨더링 노력을 돕기 위해 U.S. Census가 실제 재식별을 가능하게 하는 통계를 공개하도록 강제하는 것인지 알 수 없음
- 반대로 목표가 연구자들이 인구 내 불공정한 격차를 보이지 못하게 유용한 인구통계 데이터 공개를 막는 것인지도 알 수 없음
- [Hanlon's razor](https://en.wikipedia.org/wiki/Hanlon%27s_razor)는 대안적 해석을 제공함
- 통계 데이터 공개에는 근본적인 프라이버시/유용성 절충이 있으며, 이 절충은 성가신 문제임
- 많은 통계를 공개해도 높은 프라이버시 위험이 자동으로 따라오지 않는다면 훨씬 쉬운 상황이 됨
